## 3. 실행 모드별 데이터 흐름

### 3-1. 종목 분석 모드 ("삼성전자 분석해줘")
1. orchestrator: 종목·시장(KR/US) 확정, 실행 ID 생성 (`reports/{YYYY-MM-DD}_{티커}/`)
2. 병렬 A: 2~9번 동시 실행. 각자 결과를 `reports/.../{에이전트명}.md`로 저장
3. 7·8번은 실행 전 당일 캐시(`cache/macro-{YYYY-MM-DD}.md` 등) 확인 → 있으면 종목 연관 해석만 수행
4. red-team(10): 2~9 결과 파일을 읽고 반대 논거 생성 → 저장
5. report-writer(11): 전체 파일 읽고 최종 리포트 `report.md` 작성
6. 실패 처리: 특정 에이전트 실패 시 해당 노드만 실패 표시, 11번은 나머지로 작성 + 누락 명시

### 3-2. 종목 발굴 모드 ("요즘 살 만한 반도체주 찾아줘")
1. orchestrator: 발굴 쿼리 판별, 탐색 조건 추출 (섹터·시장·테마)
2. 1차 웨이브: 4(거장 매수 종목)·5(SNS 화제)·6(뉴스 모멘텀)·8(정책 수혜) 병렬 → 후보 리스트
3. orchestrator: 후보 중복·근거 수 기준으로 상위 2~3종목 선정
4. 2차 웨이브: 선정 종목별 2·3번 압축 검증 (풀 리포트 아님, 핵심만)
5. report-writer: 비교 리포트 (종목별 강약점, 최종 순위, 근거 각주)

### 3-3. 후속 질문 모드 ("WACC 10%로 바꿔서 다시")
1. orchestrator: 직전 실행 ID의 리포트 폴더 로드
2. 해당 질문과 관련된 에이전트만 재호출 (예: WACC → 2번만)
3. report-writer: 해당 섹션만 갱신한 증분 리포트

## 4. 에이전트 하네스 구조 (초심자용 설명 포함)

### 4-1. 하네스란
- 하네스(harness) = 에이전트가 실제로 "돌아가게" 해주는 실행 환경. 자동차에 비유하면 프롬프트는 운전 지시서, 모델은 엔진, 하네스는 차체·배선·페달
- 하네스가 하는 일: ① 모델 호출 반복 루프(생각→도구 실행→결과 회수→다시 생각) ② 도구 제공(웹 검색·파일 읽기쓰기·Bash·MCP) ③ 권한 관리 ④ 서브에이전트 생성·격리 ⑤ 진행 이벤트 스트리밍
- 본 프로젝트는 하네스를 직접 만들지 않음. 기존 하네스 2개를 단계별로 사용

### 4-2. 1단계 하네스 = Claude Code 그 자체
```
사용자 (터미널)
  └─ Claude Code  ← 하네스
      ├─ /투자분석 스킬 (오케스트레이터 지침 로드)
      ├─ .claude/agents/*.md  → Agent 툴로 서브에이전트 스폰
      │    └─ 각 서브에이전트: 독립 컨텍스트 + 자기 도구(웹검색·Bash·MCP)
      └─ 파일시스템 reports/ ← 에이전트 간 결과 전달 통로
```
- 서브에이전트는 **각자 독립된 컨텍스트 창**을 가짐 → 서로의 대화를 못 봄 → 결과 전달은 파일로 (컨텍스트 절약 + 근거 원문 보존 + 아카이브가 공짜로 생김)
- 이 단계 목적: 프롬프트 품질 튜닝. 웹 없이 터미널에서 2~3종목 돌려보며 수정

### 4-3. 2단계 하네스 = Claude Agent SDK (Node 백엔드 내장)
```
브라우저 (Vercel 정적 프론트)
  │  ① 쿼리 POST + ② SSE 구독 (Cloudflare Tunnel 주소로)
  ▼
맥의 Node 서버 (server/)
  └─ Claude Agent SDK query()  ← 하네스
      ├─ 시스템 프롬프트 = orchestrator 지침 (1단계와 동일 파일 로드)
      ├─ agents 옵션 = .claude/agents/*.md 정의 (동일 파일 재사용)
      ├─ 인증 = Claude Code 구독 로그인 재사용 (API 키 불필요)
      └─ 스트림 이벤트 (서브에이전트 시작/도구 사용/완료)
             │ 서버가 파싱·변환
             ▼
        SSE 이벤트 → 프론트 노드 캔버스 실시간 갱신
```
- SDK는 Claude Code와 동일한 엔진의 프로그래밍 버전 → **에이전트 정의 파일이 두 하네스에서 그대로 호환**되는 것이 본 설계의 축
- 서버 역할은 3가지뿐: (a) SDK 실행 (b) SDK 스트림 → SSE 이벤트 변환 (c) reports/ 아카이브 API 제공

### 4-4. 하네스 관련 리스크와 대응
- 구독 사용량: 병렬 8개 에이전트 = 순간 사용량 급증. 대응: 매크로 캐시, 발굴 모드 압축 검증, 필요 시 오케스트레이터가 "간단 분석"(2·3·6번만) 선택 가능하게 설계
- 장시간 실행: 1회 분석 5~15분. SSE 타임아웃 대응으로 서버에서 30초 간격 keep-alive 이벤트 발송
- 서브에이전트 이벤트 식별: SDK 스트림에서 서브에이전트별 이벤트를 구분해 노드 ID에 매핑 (구현 시 SDK 문서의 subagent 메시지 타입 확인 필요 — Opus 구현 시 검증 항목)

## 5. 웹사이트 스펙 (Langflow 스타일)

### 5-1. 화면 구성 (단일 페이지)
- 상단: 쿼리 입력 필드 (플레이스홀더: "종목명 또는 질문 입력 — 예: 삼성전자 분석해줘 / 요즘 살 만한 반도체주는?") + 실행 버튼
- 중앙: **노드 캔버스** (SVG). 고정 DAG 레이아웃, 라이브러리 없이 바닐라 JS
  - 종목 분석 모드: `[쿼리] → [오케스트레이터] → (2~9번 8개 노드 세로 배치) → [레드팀] → [리포트]`
  - 발굴 모드: 2단 웨이브 레이아웃 (발굴 4노드 → 선정 → 검증 2×N노드 → 리포트)
- 노드 상태: 대기(회색, hairline 테두리) / 실행 중(블루 #0075de 펄스 + 현재 작업 한 줄 실시간 갱신) / 완료(그린 체크 + 결과 3줄 요약) / 실패(오렌지 경고)
- 실행 중 연결선: 흐르는 대시 애니메이션 (CSS stroke-dashoffset)
- **노드 클릭 → 우측 사이드 패널**: 해당 에이전트 결과 md 전문 렌더 (marked.js 정도의 경량 렌더러)
- 하단: 최종 리포트 (줄글). 각주 [1]~[n] 클릭 시 해당 에이전트 사이드 패널 오픈 + 해당 위치 스크롤
- 별도 탭: 아카이브 (과거 리포트 목록 → 클릭 시 당시 노드 그래프 + 리포트 복원)

### 5-2. 디자인
- 토큰 소스: `CJM 대시보드-2026/앱개발/DESIGN-notion.md` 를 프로젝트로 복사해 사용
- 캔버스 배경 canvas-soft(#f6f5f4), 노드 카드 surface(#ffffff) + hairline(#e6e6e6) + rounded-md(8px), 본문 NotionInter/Inter, 포인트 블루 #0075de
- 에이전트별 액센트 컬러: Notion 스티커 팔레트 매핑 (다모다란=accent-purple, 밸류=accent-sky, 스마트머니=accent-green, SNS=accent-pink, 뉴스=accent-orange, 매크로=accent-teal, 정책=accent-brown, 기술적=secondary, 레드팀=accent-orange-deep)

### 5-3. 서버 ↔ 프론트 SSE 이벤트 스키마
```json
{ "type": "run_started",  "runId": "2026-07-07_AAPL", "mode": "analysis|discovery|followup", "nodes": ["damodaran", "..."] }
{ "type": "node_status",  "node": "damodaran", "status": "pending|running|done|failed" }
{ "type": "node_progress","node": "damodaran", "text": "10-K에서 세그먼트 매출 확인 중" }
{ "type": "node_result",  "node": "damodaran", "summary": "3줄 요약", "file": "reports/.../damodaran-analyst.md" }
{ "type": "report_chunk", "text": "..." }
{ "type": "run_done",     "reportFile": "reports/.../report.md" }
{ "type": "keepalive" }
```

### 5-4. API (백엔드, 맥 로컬 + Cloudflare Tunnel)
- `POST /api/run` — { query, secret } → { runId }, SSE는 `GET /api/stream/{runId}`
- `GET /api/reports` — 아카이브 목록 (index.json)
- `GET /api/reports/{runId}` — 리포트 + 에이전트별 결과
- 인증: 공유 비밀키 1개 (프론트 최초 접속 시 입력 → localStorage). CORS는 Vercel 도메인만 허용

## 7. 공통 규약 (모든 에이전트 프롬프트에 적용)

- 데이터 3단계 태깅: [실제](출처 명시) / [추정](근거 명시) / [가정](변경 가능 명시). 출처 없는 숫자에 [실제] 금지
- 모른다 선언: 확인 불가 정보는 불확실성 명시. 숫자 지어내기 금지
- 결과 저장: `reports/{YYYY-MM-DD}_{티커 또는 쿼리슬러그}/{에이전트명}.md`
- 결과 파일 구조: 상단 3줄 요약 → 본문 → 출처 목록 → 신뢰도 자가평가 1줄
- 언어: 한국어 + 재무용어 영문 병기
- 시장 구분: 오케스트레이터가 전달한 market(KR/US)에 따라 소스 분기
- 면책: 리포트 말미 "본 리포트는 투자 자문이 아니며 정보 제공 목적" 고지

## 8. 구현 로드맵 (Opus 세션용)

- Phase 0 — 준비 (사용자 액션 포함)
  - 사용자: DART·FRED·Congress.gov 키 3종 발급·검증 완료(2026-07-07). 공공데이터포털은 불채택(§6). → `.env` 저장
  - 폴더 스캐폴드: `.claude/agents/`, `.claude/skills/투자분석/`, `server/`, `web/`, `reports/`, `cache/`, `docs/`
- Phase 1 — 에이전트 (CLI 검증)
  - `docs/prompts/*.md` → `.claude/agents/*.md` 이식 (frontmatter 포함)
  - 오케스트레이터 지침 → `.claude/skills/투자분석/SKILL.md`
  - 검증: 미국 1종목(예: AAPL)·한국 1종목(예: 삼성전자)·발굴 쿼리 1건 실행, 결과 품질 확인 후 프롬프트 수정
- Phase 2 — **정적 export 파이프라인** (백엔드 폐기, 피벗 반영)
  - `tools/export.mjs`(Node, 무의존): `reports/`를 스캔 → `web/data/index.json`(아카이브 목록) + `web/data/{runId}.json`(노드 그래프 구조·에이전트별 3줄요약·상태·리포트 md) 생성
  - 노드 그래프는 존재하는 에이전트 파일로 역구성(있음=done, 없음=failed/누락), 각주는 report.md의 `[n] {에이전트명}` 파싱
  - SSE·keep-alive·비밀키·CORS 전부 불필요 (정적 파일만)
- Phase 3 — **정적 뷰어 프론트** (`web/`)
  - 무빌드 HTML/CSS/JS, Notion 토큰(`DESIGN-notion.md`), SVG 고정 DAG 노드 캔버스(완료 스냅샷), 노드 클릭→사이드 패널(에이전트 md 렌더), 하단 리포트+각주 클릭 연동, 아카이브 탭
  - md 렌더는 경량 자체 렌더러 또는 인라인 marked(정적 자산). `web/data/`를 fetch (동일 오리진 정적)
- Phase 4 — **Vercel 정적 배포**
  - `web/`(+`web/data/`) 정적 배포. cloudflared·비밀키·CORS 없음. 맥 꺼져도 열람 가능
  - 운영: CLI로 분석 → `node tools/export.mjs` → `web/` 커밋/배포(또는 로컬 열람)
- Phase 5 — 검증 시나리오
  - export 후 뷰어에서: 분석 모드 KR/US 리포트 렌더, 노드 그래프·상태 표시, 노드 클릭·각주 클릭 동작, 실패 에이전트 누락 표시, 아카이브 복원 확인

### 10-3. Rules — 프로젝트 `.claude/settings.json` 권한 설정 (병렬 8에이전트가 권한 프롬프트에 막히지 않기 위한 필수 설정)

```jsonc
{
  "permissions": {
    "allow": [
      "WebSearch",
      "WebFetch",
      "Read", "Write(reports/**)", "Write(cache/**)",
      "Bash(curl -s https://data.sec.gov/*)",
      "Bash(curl -s https://efts.sec.gov/*)",
      "Bash(curl -s https://api.usaspending.gov/*)",
      "Bash(curl -s https://api.stlouisfed.org/*)",
      "Bash(curl -s https://opendart.fss.or.kr/*)",
      "Bash(curl -s https://www.reddit.com/*)",
      "Bash(curl -s https://api.stocktwits.com/*)",
      "Bash(curl -s https://api.congress.gov/*)",
      "Bash(python3 *)"
    ],
    "deny": ["Bash(rm -rf*)", "Read(.env)", "Read(**/.env)"]
  }
}
```
- `.env` 읽기 deny + 스크립트에서 환경변수로만 주입 → 에이전트가 키 원문을 컨텍스트에 올리지 않음
- 정확한 문법은 Opus 구현 시 Claude Code 권한 문서 기준으로 조정
